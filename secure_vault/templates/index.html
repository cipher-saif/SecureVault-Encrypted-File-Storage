ℹ️ HOW IT WORKS
1
Key Derivation
Your password → PBKDF2-HMAC-SHA256 with 600,000 iterations + random 32-byte salt → AES-256 master key
2
File Key Generation
Random AES-256 file key + HMAC key generated per-file. Wrapped with master key via AES-GCM.
3
Chunk Encryption
File split into 1MB chunks. Each chunk encrypted with AES-256-GCM. Unique nonce per chunk prevents reordering attacks.
4
Integrity Seal
HMAC-SHA256 over entire payload appended. Server also stores blob hash. Two-layer integrity guarantee.
5
Secure Upload
Only ciphertext leaves your machine. Server stores encrypted blob. Password NEVER transmitted.

🔄 DECRYPTION FLOW
① INTEGRITY CHECK FIRST
Server verifies encrypted blob SHA-256 hash before transmitting. Guards against storage tampering.
② AUTHENTICATE THEN DECRYPT
HMAC verified over full payload before any decryption begins. Follows cryptographic best practice — prevents padding oracle attacks.
③ CHUNK VERIFICATION
Each chunk has its own AES-GCM authentication tag (16 bytes). Chunk order verified via AAD. Any bit flip → immediate abort.
④ SECURE CLEANUP
Temp encrypted blob overwritten with random bytes before deletion. Plaintext temp files wiped.

🛡️ THREAT MODEL & MITIGATIONS
🔴 HIGH SEVERITY THREATS
👤
Man-in-the-Middle (MITM)
An attacker intercepts traffic between client and server, potentially reading or modifying file data in transit.
✅ MITIGATED: Files are encrypted client-side with AES-256-GCM before transmission. Even if traffic is intercepted, attacker sees only ciphertext. Deploy behind HTTPS/TLS for transport-layer defense.
HIGH
🏢
Compromised Server / Insider Threat
Server is breached; attacker gains access to all stored files and metadata.
✅ MITIGATED: Zero-knowledge architecture — server stores only ciphertext. Password never transmitted. Attacker gains only useless encrypted blobs.
HIGH
🗄️
Storage Tampering
Attacker modifies encrypted blobs on disk to inject malicious content or corrupt files.
✅ MITIGATED: Two-layer integrity: (1) HMAC-SHA256 over full payload verified before decryption, (2) Server records SHA-256 of blob at upload and verifies before download. Any modification detected.
HIGH
🟡 MEDIUM SEVERITY THREATS
⚡
Replay Attack
Attacker captures a chunk and replays it to substitute for another chunk.
✅ MITIGATED: Each chunk uses a unique nonce (8 random bytes + 4-byte counter). Chunk index is included as Additional Authenticated Data (AAD) in GCM — swapping chunks invalidates the authentication tag.
MEDIUM
🔑
Key Management / Weak Passwords
Weak passwords are susceptible to brute-force or dictionary attacks against the KDF.
✅ MITIGATED: PBKDF2-HMAC-SHA256 with 600,000 iterations (NIST SP 800-132). Random 32-byte salt prevents rainbow tables. Remaining risk: human choice of weak passwords — production should enforce password policy or use random key material.
MEDIUM
⏱️
Timing Attacks on HMAC Verification
Attacker measures response time to infer correct vs. incorrect MAC bytes.
✅ MITIGATED: hmac.compare_digest() used for constant-time comparison (Python stdlib). Eliminates timing oracle.
MEDIUM
🔄
Nonce Reuse
AES-GCM is catastrophically broken if nonces are reused with the same key (two-time pad).
✅ MITIGATED: Per-file random key generation guarantees key uniqueness. Chunk nonces use 8 random bytes + counter. A separate HMAC key is also derived. Probability of collision is negligible.
MEDIUM
🟢 MITIGATED / LOW THREATS
🗑️
Data Recovery After Deletion
Deleted file data may be recoverable from disk with forensic tools.
✅ MITIGATED: 3-pass overwrite before unlink (simplified DoD 5220.22-M). Production: use SSD-aware secure erase or encrypted volumes.
LOW
💾
Plaintext in Temp Files
Unencrypted data left in temp files after processing.
✅ MITIGATED: Temp files overwritten with os.urandom() before deletion. try/finally ensures cleanup even on error.
LOW
⚠️ PRODUCTION RECOMMENDATIONS
• Deploy behind HTTPS/TLS (Let's Encrypt)
• Use HSM or AWS KMS for server key storage
• Add per-user authentication (JWT/OAuth)
• Implement rate limiting on API endpoints
• Enable audit logging for all operations
• Add file access control (ownership)
• Use Argon2id instead of PBKDF2 for KDF
• Add client-certificate mutual TLS
• Implement key rotation for stored files
• Regular integrity audits of all stored blobs
